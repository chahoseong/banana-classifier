# 인사이트

## 문서 작성 목적
이 문서는 바나나 숙성도 분류 프로젝트를 진행하면서 얻은 협업 및 개발 프로세스 인사이트를 정리한 기록이다. 단순히 어떤 문서를 작성했는지를 나열하는 것이 아니라, 왜 그 문서가 필요했고 실제로 어떤 효과를 주었는지를 다른 사람들과 공유할 수 있도록 정리하는 데 목적이 있다.

## 전체 진행 흐름
이번 프로젝트에서는 AI agent와 협업하고, 3인 팀이 짧은 시간 안에 개발을 진행해야 했기 때문에 구현보다 먼저 문서화와 합의가 필요했다. 실제 진행 흐름은 아래와 같았다.

1. PRD를 작성하여 프로젝트 목표와 범위를 명확히 정의했다.
2. 협업.md를 작성하여 역할과 협업 방식을 정리했다.
3. 협업.md에서 데이터, 모델, 환경, 인터페이스 등 협업에 필요한 세부 사항을 결정했다.
4. TRD.md를 작성하여 개발 환경과 기술 스택, 실행 구조를 정의했다.
5. 안티그라비티를 사용할 때는 Implementation Plan 문서를 작성하여 구현 전에 agent와 사용자의 생각을 맞췄다.
6. 위 문서들을 바탕으로 실제 개발 환경을 구축하고 팀원들에게 공유했다.

이 과정을 통해 구현 전에 기준을 먼저 맞출 수 있었고, 이후 작업 속도와 품질 모두 안정적으로 유지할 수 있었다.

## 단계별 인사이트

### 1. AI agent와 작업하기 위해 PRD를 먼저 작성한 것은 매우 효과적이었다
PRD를 먼저 작성하면서 프로젝트의 목표, 분류 대상, 클래스 정의, 기술 방향, 역할 구성을 한 문서에 모을 수 있었다. 이 과정은 단순한 기획 문서 작성이 아니라, 사람과 AI agent가 같은 기준으로 작업할 수 있도록 공통 언어를 만드는 과정이었다.

특히 도움이 되었던 점은 다음과 같다.

- 프로젝트의 목표와 범위가 명확해져 불필요한 작업이 줄었다.
- 숙성도 분류 기준을 미리 정의할 수 있어 데이터 라벨링 방향이 흔들리지 않았다.
- AI agent에게 요청할 때도 맥락을 반복 설명할 필요가 줄어들었다.
- 팀원 간 이해 차이를 줄이고 빠르게 합의할 수 있었다.

정리하면, PRD는 단순한 문서가 아니라 이후 모든 작업의 기준점 역할을 했다.

### 2. 역할을 설정하고 작업을 나누기 위해 협업.md를 작성한 것이 병목을 줄였다
팀 프로젝트에서는 각자 일을 시작하는 것보다, 먼저 누가 무엇을 책임질지 정하는 것이 중요했다. 협업.md를 통해 데이터, 모델, UI/통합 등 역할을 구분하고 각자의 책임 범위를 명확히 하면서 작업 충돌을 줄일 수 있었다.

이 문서가 특히 유용했던 이유는 다음과 같다.

- 각자의 작업 시작점이 분명해졌다.
- 누가 어떤 산출물을 넘겨줘야 하는지가 명확해졌다.
- 동시에 병렬 작업이 가능해졌다.
- 구현보다 소통에 쓰이는 시간을 줄일 수 있었다.

즉, 협업.md는 역할 분담표를 넘어서 협업 비용을 줄이는 운영 문서 역할을 했다.

### 3. 협업 문서에서 인터페이스와 규칙을 먼저 정한 것이 개발 효율을 높였다
협업에서 가장 중요한 것은 단순한 역할 분배가 아니라, 서로 연결되는 지점을 먼저 정하는 것이다. 이번 프로젝트에서는 협업.md에서 데이터 형식, 모델 입출력, 평가 기준, 실행 환경, 공유 방식 등을 먼저 정하면서 이후 구현이 훨씬 수월해졌다.

이 과정에서 얻은 핵심 인사이트는 다음과 같다.

- 역할 분담만으로는 충분하지 않고, 연결 규칙까지 정의해야 실제 협업이 된다.
- 데이터 형식과 모델 인터페이스를 미리 정하면 각자 구현해도 통합이 쉬워진다.
- 실험 기록 방식과 파일 공유 방법까지 정해두면 나중에 혼선이 줄어든다.
- 환경 차이로 생기는 문제를 사전에 줄일 수 있다.

결국 협업에 필요한 사항을 문서로 먼저 결정한 것은 개발 속도를 높이는 가장 현실적인 방법이었다.

### 4. TRD.md를 통해 개발 환경을 정의한 것은 재현성과 공유 측면에서 중요했다
TRD는 구현에 필요한 기술 스택과 실행 방식, 시스템 구조를 구체적으로 정의하는 문서였다. PRD가 무엇을 만들지에 대한 문서였다면, TRD는 그것을 어떤 환경과 구조로 만들지를 정리하는 문서였다.

TRD를 작성하면서 얻은 인사이트는 다음과 같다.

- Python 버전, 라이브러리, 실행 명령을 명시해 환경 재현성이 높아졌다.
- 프론트엔드와 추론 엔진을 분리한 구조를 미리 정의해 확장 가능성을 확보했다.
- 새 팀원이 참여하더라도 빠르게 환경을 맞출 수 있었다.
- 개발 후 결과물을 공유할 때도 설명 비용이 줄었다.

짧은 프로젝트일수록 문서 없이 바로 구현하고 싶어지지만, 실제로는 TRD가 있어야 시행착오를 줄일 수 있다는 점을 확인했다.

### 5. 안티그라비티의 Implementation Plan은 구현 전에 생각을 정렬하는 데 큰 도움이 되었다
안티그라비티를 사용할 때는 단순히 바로 코드를 생성하기보다, 먼저 Implementation Plan 문서를 작성하는 것이 매우 유용했다. 이 문서는 구현 전에 무엇을 만들지, 어떤 순서로 만들지, 어떤 가정을 두고 있는지를 정리하는 역할을 한다.

특히 중요한 점은 이 문서가 agent와 사용자의 생각을 먼저 맞추는 장치로 작동한다는 것이다.

- 구현 전에 범위와 우선순위를 다시 확인할 수 있다.
- agent가 이해한 요구사항과 사용자가 의도한 바의 차이를 조기에 발견할 수 있다.
- 피드백을 문서 단계에서 반영할 수 있어 코드 수정 비용이 줄어든다.
- 구현 착수 전에 작업 단위를 나눌 수 있어 진행 관리가 쉬워진다.

즉, Implementation Plan은 설계 문서이면서 동시에 커뮤니케이션 문서였다. 구현 전에 한 번 더 생각을 맞추는 과정이 전체 품질을 높여 주었다.

### 6. 안티그라비티가 생성하는 다양한 문서는 협업과 추적 가능성을 높여 주었다
안티그라비티는 단순히 코드만 만드는 도구가 아니라, 작업을 여러 문서 단위로 구조화할 수 있게 해준다는 점에서 의미가 있었다. 예를 들어 Task, Walkthrough, Implementation Plan 같은 문서는 각각 역할이 다르며 함께 사용할수록 효과가 커졌다.

각 문서의 의미는 다음과 같이 정리할 수 있다.

- `Implementation Plan`:
  구현 전에 목표, 범위, 순서, 가정을 정리하는 문서. 사용자와 agent의 생각을 맞추는 데 특히 중요하다.
- `Task`:
  해야 할 일을 작게 나누어 관리하는 문서. 작업 우선순위를 분명히 하고 진행 상황을 추적하는 데 유용하다.
- `Walkthrough`:
  구현 결과나 구조를 설명하는 문서. 다른 팀원이나 외부 공유 대상이 전체 흐름을 빠르게 이해하는 데 도움이 된다.

이런 문서들이 있으면 다음과 같은 장점이 있다.

- 작업의 시작 전, 진행 중, 완료 후 상태를 각각 다른 문서로 명확히 설명할 수 있다.
- 팀원이나 새로운 참여자가 프로젝트 문맥을 더 빨리 이해할 수 있다.
- AI agent가 어떤 기준으로 작업했는지 추적하기 쉬워진다.
- 결과만 남는 것이 아니라, 의사결정 과정까지 공유할 수 있다.

결국 안티그라비티가 생성하는 다양한 문서는 코드 생산성뿐 아니라 협업 투명성과 재사용성을 높이는 데 기여했다.

### 7. 문서를 기반으로 개발 환경을 구축하고 공유한 것이 팀 속도를 올렸다
문서 작성이 끝난 뒤 바로 개발 환경을 구축하고 팀원들과 공유한 과정도 중요했다. 문서만 있고 실제 실행 환경이 정리되지 않으면 팀원별로 다른 방식으로 세팅하게 되고, 작은 차이가 큰 문제로 이어질 수 있다.

이번 과정에서 확인한 점은 다음과 같다.

- 문서와 실제 환경은 함께 관리되어야 한다.
- 환경을 먼저 맞추면 이후 논의가 코드와 결과 중심으로 바뀐다.
- 팀원 간 질문이 줄고, 바로 구현에 들어갈 수 있다.
- AI agent와 협업할 때도 현재 구조를 기준으로 빠르게 수정과 자동화를 진행할 수 있다.

즉, 문서화의 가치는 문서를 쓰는 데서 끝나는 것이 아니라, 실제 실행 가능한 환경으로 연결될 때 완성된다.

## 이번 프로젝트에서 얻은 핵심 교훈
이번 경험을 통해 얻은 가장 중요한 교훈은 다음과 같다.

- AI agent와 협업할수록 사전 문서화의 중요성이 더 커진다.
- 팀 프로젝트에서는 구현보다 먼저 기준과 책임을 맞추는 일이 중요하다.
- PRD, 협업 문서, TRD는 각각 역할이 다르며 서로 보완적이다.
- 협업 문서는 역할 분배뿐 아니라 인터페이스와 규칙 정의까지 포함해야 한다.
- 기술 문서는 재현성과 공유 가능성을 높여 준다.
- Implementation Plan은 구현 전에 agent와 사용자의 생각을 정렬하는 데 매우 효과적이다.
- Task, Walkthrough 같은 보조 문서는 작업 추적과 공유 품질을 높여 준다.
- 짧은 일정일수록 먼저 정리하고 나중에 구현하는 방식이 오히려 더 빠르다.

## 다른 팀에게 공유할 수 있는 권장 프로세스
비슷한 프로젝트를 진행하는 팀에게는 다음 순서를 권장할 수 있다.

1. PRD를 작성해 목표와 범위를 먼저 고정한다.
2. 협업 문서에서 역할, 책임, 인터페이스, 실험 규칙을 정한다.
3. TRD로 기술 스택과 실행 환경을 명시한다.
4. 안티그라비티를 사용할 경우 Implementation Plan을 작성해 구현 전 피드백을 반영한다.
5. Task 단위로 작업을 쪼개고, 필요할 때 Walkthrough 문서로 구조와 결과를 공유한다.
6. 실제 개발 환경을 구축하고 팀 전체에 공유한다.
7. 그 다음부터 구현과 실험을 병렬로 진행한다.

이 순서를 따르면 협업 충돌, 환경 문제, 책임 불명확성, 중복 작업을 크게 줄일 수 있다.

## 결론
이번 프로젝트는 단순히 바나나 숙성도 분류 모델을 만드는 작업이 아니라, AI agent와 사람, 그리고 팀 협업이 어떻게 연결되어야 하는지를 경험한 과정이었다. 결과적으로 문서를 먼저 정리하고, 역할과 규칙을 합의하고, 기술 환경을 명시하고, 구현 전에 다시 한 번 계획 문서로 생각을 맞춘 뒤 개발로 넘어간 방식이 매우 효과적이었다.

가장 큰 인사이트는 다음 한 문장으로 정리할 수 있다.

문서를 먼저 맞추면 사람끼리도 빠르게 협업할 수 있고, AI agent와도 훨씬 정확하고 효율적으로 일할 수 있다.
