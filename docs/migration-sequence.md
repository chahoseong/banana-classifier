# Migration Sequence

## 1. 문서 목적

이 문서는 기존 `banana-classifier` 저장소를 변경된 PRD에 맞는 **작업 환경**으로 정비하는 순서를 정리한다.

여기서 말하는 마이그레이션 범위는 다음에 한정한다.

- 저장소 구조 정리
- 레거시 자산 보존 및 분리
- 새 PRD 기준 디렉터리와 공용 규칙 정리
- 팀원들이 병렬로 작업할 수 있는 개발 환경 준비
- 데이터, 모델, 문서 자산의 위치와 책임 정리

이 문서는 새로운 기능 자체를 구현하는 계획서는 아니다.

## 2. 마이그레이션 원칙

- 기존 산출물은 삭제보다 분리와 보존을 우선한다.
- 새 PRD에 맞는 작업 환경을 먼저 만들고, 실제 구현은 그 다음 단계에서 진행한다.
- 4클래스(`unripe`, `ripe`, `overripe`, `dispose`) 체계를 유지한다.
- 프론트엔드, 백엔드, ML 자산의 책임을 분리한다.
- 팀원별 작업 충돌을 줄이기 위해 디렉터리와 인터페이스 경계를 먼저 확정한다.

## 3. 마이그레이션 범위

### 포함

- 폴더 구조 개편
- 레거시 코드 위치 정리
- 공용 개발 규칙 문서화
- 실행 환경 분리
- 모델/데이터/문서 자산 재배치
- 새 PRD 기준 상수와 정책 정리

### 제외

- React UI 구현
- FastAPI API 구현
- 새 feature extraction 구현
- 새 모델 학습 및 성능 개선
- 실시간 추론 로직 구현

## 4. 단계별 마이그레이션

### Phase 0. 현황 고정

목표:

- 현재 저장소 상태를 보존하고 이후 정리 작업의 기준점을 만든다.

작업:

1. 현재 주요 파일과 폴더를 목록화한다.
2. 기존 실행 진입점과 문서를 정리한다.
3. 현재 모델 파일, `features.csv`, 리포트 이미지 등 주요 산출물 위치를 기록한다.
4. 4클래스 정책과 기존 실험 자산 범위를 문서화한다.

산출물:

- 기준점 메모
- 보존 대상 목록

### Phase 1. 루트 구조 정비

목표:

- 새 PRD 기준 작업 공간을 저장소 차원에서 준비한다.

작업:

1. `frontend/` 디렉터리 생성
2. `backend/` 디렉터리 생성
3. `ml/` 디렉터리 생성
4. `legacy/` 디렉터리 생성
5. 필요 시 `docs/` 디렉터리 생성
6. 루트 문서의 역할을 정리한다

산출물:

- 새 루트 구조
- 디렉터리별 책임 구분

### Phase 2. 레거시 자산 분리

목표:

- 기존 코드가 새 구조와 혼재되어 팀 작업을 방해하지 않도록 정리한다.

작업:

1. 기존 Streamlit UI 관련 파일을 `legacy` 성격으로 분류한다.
2. 기존 딥러닝 중심 코드와 새 PRD의 우선순위가 낮은 코드를 `legacy` 후보로 분류한다.
3. 기존 문서 중 현재 PRD 기준과 다른 문서를 별도로 구분한다.
4. 레거시 자산은 삭제하지 않고 이동 또는 보존 계획만 확정한다.

산출물:

- 레거시 분류 목록
- 이동 대상 목록

### Phase 3. 데이터와 모델 자산 위치 정리

목표:

- 데이터셋과 모델 파일을 팀이 혼동 없이 사용할 수 있도록 기준 위치를 확정한다.

작업:

1. `dataset/raw`와 `dataset/processed`의 유지 방식을 정리한다.
2. 4클래스 데이터 정책을 문서화한다.
3. 학습 산출물 저장 위치를 `ml/artifacts/` 중심으로 정리한다.
4. 기존 모델 파일과 새 작업용 모델 파일의 네이밍 규칙을 정한다.
5. `features.csv` 같은 중간 산출물의 보관 위치를 정한다.

권장 규칙:

- 데이터셋은 `dataset/` 아래 유지
- 모델 아티팩트는 `ml/artifacts/`로 정리
- 학습용 스크립트는 `ml/training/`로 정리

산출물:

- 데이터/모델 저장 규칙
- 파일 배치 기준

### Phase 4. 공용 상수와 정책 정리

목표:

- 팀원들이 서로 다른 기준으로 구현하지 않도록 공통 전제를 먼저 맞춘다.

작업:

1. 서비스 클래스 목록을 4클래스로 확정한다.
2. 입력 이미지 기준 해상도를 `300x300`으로 문서화한다.
3. 상태명 표기 규칙을 정한다.
4. 모델 응답 기본 필드 이름을 문서화한다.
5. 향후 구현 시 따라야 할 공용 네이밍 규칙을 정한다.

정리 대상 예시:

- 클래스명: `unripe`, `ripe`, `overripe`, `dispose`
- 입력 크기: `300x300`
- 공통 용어: `predict`, `confidence`, `status`

산출물:

- 공용 규칙 메모
- 팀 간 인터페이스 초안

### Phase 5. 디렉터리별 작업 경계 설정

목표:

- 각 팀원이 어느 디렉터리에서 작업해야 하는지 명확히 한다.

작업:

1. `frontend/`는 UI와 브라우저 처리 전용으로 정의한다.
2. `backend/`는 API와 서버 로직 전용으로 정의한다.
3. `ml/`은 학습, feature schema, 모델 아티팩트 관리 전용으로 정의한다.
4. `legacy/`는 참고용 코드와 이전 구현 보관용으로 정의한다.
5. 공용으로 참조할 문서는 `docs/` 또는 루트 문서로 모은다.

산출물:

- 팀별 작업 영역 정의
- 충돌 방지 가이드

### Phase 6. 실행 환경 정리

목표:

- 새 구조에서 팀원들이 각자 독립적으로 개발할 수 있는 환경을 준비한다.

작업:

1. 프론트엔드 의존성 관리 위치를 `frontend/`로 분리한다.
2. 백엔드 의존성 관리 위치를 `backend/`로 분리한다.
3. ML 관련 의존성은 `ml/` 또는 루트 정책으로 정리한다.
4. 기존 `requirements.txt`의 역할을 재정의한다.
5. 팀원이 공통으로 써야 할 실행 명령 형식을 문서화한다.

주의:

- 이 단계에서는 실제 앱 구현보다 환경 분리와 실행 규칙 정리가 우선이다.

산출물:

- 환경 구성 원칙
- 의존성 관리 정책

### Phase 7. 진입점과 문서 정리

목표:

- 저장소를 처음 여는 팀원도 현재 구조를 빠르게 이해할 수 있게 만든다.

작업:

1. README를 새 구조 기준으로 갱신한다.
2. 기존 진입점과 새 작업 디렉터리를 구분해서 설명한다.
3. 문서 간 우선순위를 정한다.
4. 레거시 문서와 현재 문서를 명확히 구분한다.

권장 문서:

- 저장소 개요
- 디렉터리 책임
- 개발 시작 방법
- 레거시 위치
- 팀 작업 규칙

산출물:

- 새 구조 기준 README
- 문서 인덱스

## 5. 추천 작업 순서 요약

아래 순서로 진행하는 것이 가장 안전하다.

1. 현재 자산과 기준점 기록
2. 새 루트 구조 생성
3. 레거시 자산 분류
4. 데이터/모델 저장 위치 정리
5. 공용 상수와 정책 문서화
6. 팀별 작업 경계 확정
7. 의존성 및 실행 환경 분리
8. README와 문서 정리

## 6. 팀 협업 관점의 완료 기준

마이그레이션 완료는 아래 상태를 의미한다.

- 저장소 구조가 새 PRD 기준으로 정리되어 있다.
- 기존 코드와 레거시 자산의 위치가 명확하다.
- 4클래스 정책이 문서에 반영되어 있다.
- 프론트엔드, 백엔드, ML 작업 디렉터리가 구분되어 있다.
- 데이터와 모델 파일의 저장 위치가 확정되어 있다.
- 팀원들이 서로 충돌 없이 병렬 작업을 시작할 수 있다.

## 7. 마이그레이션 이후 단계

이 문서의 범위는 여기까지다.

다음 단계인 실제 구현 작업은 별도 문서나 팀별 계획에서 다룬다.

- 프론트엔드 구현 계획
- 백엔드 구현 계획
- ML 개선 및 재학습 계획
- 통합 테스트 계획
