# FE-BE 협업 가이드라인

본 문서는 실시간 바나나 숙성도 판별 서비스(Banana Ripe Checker)의 원활한 구현을 위해, 백엔드(FastAPI) 개발자와 프론트엔드(React) 개발자가 사전에 논의되어야 했던 내용들을 **협의 완료(결정)된 상태**로 정리한 기술 가이드라인입니다. `banana-ripe-checker-prd.md`의 기획 요구사항을 기준으로 작성되었습니다.

---

## 1. 이미지 캡처 및 전송 규격
* **해상도 일치 (결정)**: 원활한 ML 전처리와 성능 최적화를 위해 Canvas에서 프레임 캡처 시 **반드시 `300x300` 해상도로 고정**하여 추출해야 합니다.
* **전송 포맷 (결정)**: 캡처된 이미지는 바이너리 파일 형태로 담아 `multipart/form-data` 형식으로 API에 전송하며, 파일 용량을 줄이기 위해 포맷은 **`image/jpeg` (quality 0.7~0.8 권장)**로 통일합니다.
* **Key Name (결정)**: 폼 데이터 전송 시 파일 파라미터(Key) 이름은 반드시 **`file`** 로 확정하여 사용합니다. (예: `formData.append('file', blob);`)

## 2. API 엔드포인트 및 응답 규격
백엔드의 기본 추론 엔드포인트 규격은 다음과 같이 확정합니다:
* **Endpoint (결정)**: `POST /predict`
* **Response Format (결정)**: 
  
프론트엔드에서는 200 OK 응답 수신 시, 다음 4가지 핵심 필드가 포함된 JSON을 파싱하여 UI 상태에 매핑합니다.
```json
{
  "is_banana": true,         // 문지기 결과 (true: 바나나 맞음, false: 배경 또는 다른 물체)
  "status": "ripe",          // 4단계 숙성도 (unripe, ripe, overripe, dispose)
  "confidence": 0.87,        // 예측 확신도 (0.0 ~ 1.0)
  "message": "바로 먹기 좋은 상태입니다." // 클라이언트 노출용 상태 메시지
}
```

## 3. 데이터 최적화 (UI 가이드라인 확정)
* 백엔드는 조명 밝기에 강건한 **HSV 색 공간** 변환에 크게 의존합니다.
* **가이드라인 UI 제공 (결정)**: 화면 캡처 시 바나나가 가급적 300x300 캔버스의 중앙에 크게 오도록 유도하는 **뷰파인더(사각형 가이드라인 UI) 레이어를 카메라 영상 위에 의무적으로 적용**하여 정확도를 보장합니다.

## 4. 실시간 전송 로직 제어 (Throttle & Error Handling 확정)
가장 중요한 네트워크 부하 방지 및 지연(Latency) 관리 규약입니다.
* **이전 요청 응답 수신 대기 (결정)**: 무조건적인 고정 주기의 `setInterval` 전송 방식을 허용하지 않습니다. 프레임을 전달한 후 **서버의 완료 응답(200 OK 등)이 수신된 이후에만 다음 프레임을 추출/전송하는 재귀적 호출 트리를 강제**합니다.
* **클라이언트 로컬 대기 시간 (결정)**: 백엔드 응답을 받은 시점으로부터 **정확히 최소 500ms(로컬 스로틀) 이상 대기**한 후 다음 트랜잭션을 발생시켜 서버 부하를 조절합니다.
* **예외/오류 처리 (결정)**: `4xx`, `5xx` 통신 에러나 네트워크 TimeOut 발생 시, 화면 마비 현상을 막기 위해 **3초(3000ms) 대기 후 재시도(Retry)** 하는 로직을 구비하고, 사용자에게 화면 중앙에 "서버와 연결을 재시도합니다" 같은 에러 토스트를 표출합니다.

## 5. UI 상태 분류 및 피드백 매핑 (결정)
반환받은 숙성도를 토대로 바운딩 박스 색상(또는 메인 테마 색상)을 즉각 변경하는 규칙입니다.
* **`is_banana: false` 일 때**:
  * **회색 (Gray)** 테마 및 "바나나 영역을 맞춰주세요" 안내 노출
* **`is_banana: true` 이면서 `status` 변화 시**:
  1. `unripe`: **녹색 (Green)** 계열 (미숙)
  2. `ripe`: **노란색 (Yellow)** 계열 (적숙)
  3. `overripe`: **갈색 (Brown)** 계열 (과숙/베이킹용)
  4. `dispose`: **검정색 또는 빨간색 (Black/Red)** 계열 (부패/버림)

위 결정 사항들에 맞춰 FE 및 BE 로직을 동시 다발적으로 즉각 구현하며, 명세에 벗어나는 오류나 추가 필드 요구가 필요해질 시 즉각 양측 간 협의를 거쳐 API 및 클라이언트 로직을 갱신합니다.
