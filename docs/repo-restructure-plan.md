# Repository Restructure Plan

## 1. 문서 목적

이 문서는 기존 `banana-classifier` 저장소를 변경된 PRD에 맞게 **정비된 작업 공간**으로 재구성하는 기준을 정리한다.

이 문서의 초점은 다음과 같다.

- 현재 저장소를 유지할지에 대한 구조적 판단
- 어떤 자산을 유지하고 어떤 자산을 분리할지에 대한 기준
- 팀원들이 병렬로 작업할 수 있도록 디렉터리 책임을 나누는 방법
- 새 PRD에 맞는 작업 환경으로 저장소를 정리하는 방향

이 문서는 기능 구현 상세 설계서가 아니라, 저장소 구조와 협업 준비를 위한 기준 문서다.

## 2. 개편 방향

기본 방향은 `새 프로젝트 생성`이 아니라 `기존 저장소 개편`이다.

그 이유는 다음과 같다.

- 데이터셋과 전처리 자산이 이미 존재한다.
- kNN 중심 학습 자산과 feature extraction 코드가 남아 있다.
- 기존 실험 결과와 문서를 버리지 않고 보존할 수 있다.
- 팀원별 역할에 맞춰 디렉터리만 재정리해도 새 PRD 대응이 가능하다.

따라서 저장소는 유지하되, 새 PRD에 맞는 작업 경계와 자산 위치를 다시 정한다.

## 3. 개편 원칙

- 기존 산출물은 삭제보다 보존과 분리를 우선한다.
- 4클래스(`unripe`, `ripe`, `overripe`, `dispose`) 체계를 유지한다.
- 프론트엔드, 백엔드, ML 자산의 책임을 분리한다.
- 기존 Streamlit 및 이전 실험 코드는 `legacy`로 분리한다.
- 데이터셋, 모델, 문서는 위치와 책임이 바로 이해되도록 재배치한다.
- 팀원이 처음 저장소를 열어도 현재 기준 구조를 이해할 수 있어야 한다.

## 4. 유지할 자산

다음 자산은 새 구조에서도 계속 활용할 가치가 높다.

- `dataset/`
  - 원본 및 가공 이미지 데이터 보관 자산
- `data/preprocess.py`
  - 데이터 전처리 참고 자산
- `data/extract_features.py`
  - feature extraction 참고 자산
- `src/preprocessing/image_processor.py`
  - 추론/전처리 공용 로직 후보
- `train_knn.py`
  - ML 학습 파이프라인 참고 자산
- `src/inference/inference_engine.py`
  - 모델 분리 구조 참고 자산
- `models/`
  - 저장 모델 및 기존 결과물 보관 자산

이 자산들은 새 구조에서 그대로 쓰거나, 위치를 바꿔 재정리하는 대상이다.

## 5. 분리할 자산

다음 자산은 즉시 삭제하지 않고 `legacy` 성격으로 분리하는 편이 좋다.

- `src/app/main.py`
  - 기존 Streamlit/WebRTC UI
- `main.py`
  - 현재 실행 진입점 중 기존 구조에 가까운 코드
- `models/custom_dl.py`
  - 현재 우선순위가 낮은 딥러닝 실험 자산
- `train.py`, `evaluate.py`
  - 새 작업 환경 기준에서는 중심 경로가 아닌 자산
- 기존 문서 중 이전 PRD 전제에 묶여 있는 문서들

분리 기준은 단순하다.

- 새 PRD 기준 작업 환경의 중심이 아닌 코드
- 팀원들이 새 구조를 이해하는 데 혼선을 줄 수 있는 코드
- 당장 구현 시작점으로 쓰이지 않을 코드

## 6. 권장 디렉터리 구조

아래와 같이 큰 책임 단위로 정리하는 것을 권장한다.

```text
banana-classifier/
├── backend/
├── frontend/
├── ml/
├── dataset/
├── legacy/
├── docs/
├── scripts/
├── .gitignore
└── README.md
```

이 구조의 목적은 상세 구현을 미리 고정하는 데 있지 않다.

핵심은 다음 네 가지를 분리하는 것이다.

- 사용자 인터페이스 작업 공간
- 서버/API 작업 공간
- 모델 학습 및 아티팩트 작업 공간
- 이전 코드와 문서 보관 공간

## 7. 디렉터리별 책임

### `frontend/`

- 브라우저 기반 UI 작업 공간
- 카메라 처리 및 사용자 화면 관련 코드 위치
- 프론트엔드 의존성 관리 위치

### `backend/`

- API 및 서버 로직 작업 공간
- 백엔드 의존성 관리 위치
- 모델 호출과 서비스 계층 연결 위치

### `ml/`

- 학습 스크립트 정리 위치
- feature schema 및 모델 관련 실험 자산 관리 위치
- 모델 아티팩트 저장 위치

### `dataset/`

- 원본 데이터와 가공 데이터 위치
- 팀 전체가 공통으로 참조하는 데이터 기준 위치

### `legacy/`

- 기존 Streamlit 앱
- 기존 딥러닝 실험 코드
- 이전 구조 기준 문서와 참고 코드

### `docs/`

- 현재 기준 문서 보관 위치
- 구조, 협업 규칙, 실행 방법, 작업 시작 가이드 정리 위치

## 8. 공통 정책

### 클래스 정책

서비스와 학습 모두 기존 4클래스 체계를 유지한다.

- `unripe`
- `ripe`
- `overripe`
- `dispose`

### 입력 규격 정책

새 PRD 기준 입력 규격은 `300x300`으로 통일하는 것을 권장한다.

이 정책은 다음 항목에 공통으로 적용된다.

- 데이터 전처리 기준
- 프론트엔드 캡처 기준
- 백엔드 입력 기준
- 추후 모델 관련 공용 상수

### 자산 관리 정책

- 데이터셋은 `dataset/` 아래에서 관리한다.
- 모델 파일은 `ml/` 아래에서 관리한다.
- 레거시 코드는 `legacy/` 아래에서 관리한다.
- 현재 기준 문서는 `docs/` 또는 루트 기준 문서로 관리한다.

## 9. 파일 재배치 기준

### 새 구조로 옮겨 정리할 후보

- 전처리 관련 공용 코드
- feature extraction 관련 코드
- kNN 학습 스크립트
- 모델 저장 파일

### 참고용으로 남길 후보

- Streamlit 앱 코드
- 기존 딥러닝 실험 코드
- 이전 PRD 기반 문서

### 옮기기 전에 기준을 먼저 정할 항목

- 공용 상수
- 의존성 파일 위치
- 모델 파일 네이밍 규칙
- 문서 우선순위

## 10. 협업 관점에서 기대 효과

이 구조 개편이 끝나면 다음 효과를 기대할 수 있다.

- 팀원이 각자 작업할 위치가 명확해진다.
- 이전 코드와 새 작업 코드가 섞이지 않는다.
- 데이터, 모델, 문서의 기준 위치가 분명해진다.
- 새 PRD 대응 작업을 병렬로 시작하기 쉬워진다.
- 기존 산출물을 잃지 않고 새 구조로 전환할 수 있다.

## 11. 최종 결론

현재 저장소는 폐기보다 개편이 적절하다.

정확히는 다음 방식이 가장 현실적이다.

- 저장소는 유지한다.
- 자산은 유지하되 책임별로 재배치한다.
- 기존 앱과 실험 코드는 `legacy`로 분리한다.
- 새 PRD 대응을 위한 작업 공간은 `frontend`, `backend`, `ml`로 분리한다.
- 문서는 현재 기준과 레거시 문서를 분리해 관리한다.

이 방향이 기존 작업을 최대한 살리면서도, 팀이 새 PRD 기준으로 안정적으로 협업을 시작하기에 가장 적합하다.
