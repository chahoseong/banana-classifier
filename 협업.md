# 팀원 1(데이터 엔지니어) & 팀원 2(모델 엔지니어) 협업 합의 사항

본 문서는 `PRD.md`에 명시된 역할에 따라 **팀원 1(데이터 엔지니어 및 베이스라인 개발)**과 **팀원 2(딥러닝 모델 개발)**가 원활하게 협업하기 위해 사전에 합의된 기술적/절차적 사항을 정의합니다.

---

## 1. 데이터 입출력 인터페이스 (Data I/O)

데이터 엔지니어가 구축한 파이프라인의 결과물이 모델 엔지니어의 입력으로 일관되게 제공되도록 규격을 확정합니다.

*   **이미지 규격 (Resolution):**
    *   224 x 224 픽셀 (ResNet 등 기성 모델의 표준 규격 활용)
*   **전처리 방식 적용 시점:**
    *   해상도 통일(비율 유지 패딩 및 리사이즈)과 기본 형태 정제는 **미리 진행하여 디스크에 저장**.
    *   정규화(Normalization, ImageNet 평균/표준편차 적용) 및 데이터 증강(Augmentation)은 PyTorch의 `DataLoader`에서 **On-the-fly(실시간)**로 처리.
*   **데이터 세트 분할 (Train/Val/Test Split):**
    *   비율: 8 : 1 : 1
    *   재현성을 위한 Random Seed: `42`로 고정.
*   **데이터 저장 형식:**
    *   전처리가 완료된 파일은 **구조화된 디렉토리 형태의 이미지 파일(`.jpg`)**로 저장하여 직관성을 높이고 PyTorch `ImageFolder` 방식을 쉽게 사용할 수 있도록 함.

## 2. 모델 인터페이스 규격 (Base Model Interface)

다중 모델 지원 방식(ML, DL 등) 및 일관성 있는 예측 파이프라인을 위해 인터페이스 구조를 통일합니다.

*   **입력(Input) 형태:**
    *   배치(Batch) 차원이 포함된 PyTorch Tensor 형태 `(B, C, H, W)` (RGB, 정규화된 float32).
    *   단일 이미지 추론 시에도 형태를 맞추기 위해 배치 차원 `B=1`을 추가하여 입력.
*   **출력(Output) 형태:**
    *   단순 클래스 인덱스가 아닌 모델의 **예측 확률 배열(Softmax)** 또는 **로짓(Logits)** 반환.
    *   예: `(B, 4)` 형태의 Tensor 반환. 추론 코드에서 이 값을 바탕으로 `argmax`를 적용해 최종 클래스를 찾고, 최대 확률값을 신뢰도(Confidence)로 사용.
*   **클래스 매핑(Class Mapping):**
    *   `{0: 'unripe', 1: 'ripe', 2: 'overripe', 3: 'dispose'}`

## 3. 평가 지표 및 측정 기준 (Evaluation Metrics)

목표 달성 여부와 두 사람이 개발한 모델 간 객관적인 성능 비교를 위해 평가 기준을 동기화합니다.

*   **기본 평가 지표 (Metrics):**
    *   전체 분류 정확도(Accuracy)
*   **추가 지표 (클래스 불균형 고려):**
    *   F1-Score (Macro), Precision, Recall
    *   Confusion Matrix를 함께 산출하여 특정 숙성도의 오분류 패턴 파악
*   **추론 속도(FPS) 측정 환경:**
    *   측정 기준: 전처리 로직 시간과 모델 추론 시간을 모두 더한 **전체 파이프라인 속도** 기준.
    *   환경: 노트북 CPU 환경(Intel i5/Ryzen 5 상당)에서 모델 추론이 장당 1초 이내로 완료되는지 확인.

## 4. 데이터 증강 (Data Augmentation) 역할 분담

데이터 파이프라인과 모델 학습 파이프라인 간 작업 충돌을 방지합니다.

*   **팀원 1 (전처리 및 파이프라인 담당):**
    *   기하학적 보정(비율 유지 패딩), 배경 및 노이즈 제거 등 **데이터 세트 원본에 대한 정적 전처리** 담당.
*   **팀원 2 (모델링 및 학습 담당):**
    *   학습 성능과 일반화 능력 향상을 위한 Random Flip, Rotation, Color Jitter 등 **학습 과정(Tensor 변환 단계)에서의 확률적 Data Augmentation** 담당.

## 5. 코드 컨벤션 및 관리 전략 (Git / Repository)

*   **환경 설정 통일:**
    *   `requirements.txt` 관리. 새로운 라이브러리 추가 시 Commit 내용에 명시하고 톡/게시판을 통해 즉시 공유.
*   **모델 가중치(Weight) 공유 방식:**
    *   용량이 큰 모델 파일(`.pth` 등)은 Git Repository에 올리지 않고 **구글 드라이브 공유 폴더** 등 클라우드를 활용해 교환.
*   **가중치 파일 네이밍 규칙 합의:**
    *   `[모델명]_ep[학습에폭]_acc[검증정확도]_[MMDD].pth`
    *   예) `resnet18_ep20_acc95_0312.pth`
*   **실험 기록 (Experiment Tracking):**
    *   하이퍼파라미터 및 주요 성능 결과는 **팀 노션(Notion)의 실험 기록표(스프레드시트)**에 수기로 기록하여 간편하고 빠르게 공유.

## 6. 학습 환경 분리 및 로컬/Colab 활용 (Execution Environment)

VRAM 등 하드웨어 리소스 한계를 고려하여, 파이프라인 검증과 본 학습 환경을 아래와 같이 분리하여 진행합니다.

*   **로컬 환경 (현재 노트북 CPU/보급형 GPU):**
    *   **목적:** 데이터 전처리 로직 및 모델 파이프라인에 버그가 없는지 Test (Sanity Check).
    *   **방식:** 매우 적은 데이터 셋(또는 1~2 Epoch)과 MobileNet V2 / ResNet-18 등 가벼운 모델을 활용하여 로직만 검증.
*   **클라우드 환경 (Google Colab T4 GPU 등):**
    *   **목적:** 실제 모델 본 학습(Full Training) 및 무거운 스케일 모델(ResNet-50 등) 실험.
    *   **방식:** 구글 드라이브(팀 폴더)에 압축된(`zip`) 이미지 데이터셋을 업로드한 뒤, Colab에서 마운트하여 압축을 해제하고 GPU 가속을 활용해 전체 학습 진행. 훈련 완료된 최종 `.pth` 파일만 다운받아 로컬 시스템에서 추론/서비스 연동.
